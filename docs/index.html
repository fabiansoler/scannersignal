Scanner de Señales

Plataforma web de detección automática de setups de trading en tiempo real para scalping — Cripto · Forex · Futuros

Node.js + Express WebSocket React + Vite Tailwind CSS SQLite Telegram Alerts

¿Qué es?

El Scanner de Señales es una aplicación full-stack que monitorea múltiples pares de trading simultáneamente, aplica análisis técnico sobre datos OHLCV en tiempo real y calcula un score de confluencia que indica la fuerza de cada setup detectado.

El sistema escanea 13 pares cada 10 segundos (configurable), cubre 3 mercados y soporta timeframes M1, M5 y M15. Cuando se detecta una señal fuerte (score ≥ 80), envía una alerta automática por Telegram.

Incluye además un módulo de Confluencia Multi-Timeframe: una pantalla aparte que toma un único par y analiza su confluencia de indicadores en M1, M5 y M15 simultáneamente, con un score firmado de −100 a +100 y señal STRONG/WEAK BUY/SELL en tiempo real.

Flujo de datos

1. Ciclo de escaneo (cada 10 segundos)

// scanner.js — simplificado
async function runScan(timeframe) {
  for (par of todosLosPares) {
    const candles  = await feed.fetchOHLCV(par, timeframe, 100)
    const indicators = calculateIndicators(candles)
    const { score, direction, setup } = calcScore(indicators)

    if (score >= MIN_SCORE_PUBLISH) {
      guardarEnEstado(par, { score, direction, setup })
      if (cambió) broadcastWS()
      if (score >= MIN_SCORE_ALERT) sendTelegram()
    }
  }
}

2. Push por WebSocket

Solo se envían actualizaciones cuando el score o la dirección cambian respecto al ciclo anterior. Esto evita saturar el cliente con mensajes idénticos.

Motor de escaneo

El scanner corre en un setInterval configurable (default 10s). Todos los pares se procesan en paralelo con Promise.allSettled — si un par falla, los demás continúan sin interrupción.

Indicadores técnicos

Todos los indicadores se calculan sobre las últimas 100 velas OHLCV del par. Cada uno retorna { signal, value, active }.

RSI (14 períodos)

Mide el momentum relativo de subidas vs bajadas.

CondiciónSeñalInterpretación
RSI < 35Bullish ✓Activo — zona de sobreventa, posible rebote alcista
RSI > 65Bearish ✓Activo — zona de sobrecompra, posible corrección
35 ≤ RSI ≤ 65NeutralInactivo — no aporta puntos al score

EMA (9 vs 21 períodos)

Compara la posición relativa de dos medias exponenciales. Una EMA rápida por encima de la lenta indica tendencia alcista activa.

CondiciónSeñalInterpretación
EMA9 > EMA21 (separación > 0.01%)Bullish ✓Tendencia alcista activa
EMA9 < EMA21 (separación > 0.01%)Bearish ✓Tendencia bajista activa
EMA9 ≈ EMA21NeutralMercado lateral — señal inactiva

MACD (12, 26, 9)

El histograma mide la divergencia entre la línea MACD y su señal. Se activa cuando el histograma es positivo y creciente (momentum alcista) o negativo y cayendo (momentum bajista).

Volumen relativo

Compara el volumen de la vela actual contra el promedio de las últimas 20 velas. Un spike de volumen amplifica las otras señales sin determinar la dirección por sí solo.

CondiciónActivoEfecto en el score
Volumen actual > 1.5× promedio 20 velasSíSuma 20 pts al bando dominante (amplificador)
Volumen normalNoNo aporta puntos

Score de confluencia

El score combina los indicadores activos ponderados. A mayor score, mayor confluencia de señales apuntando en la misma dirección.

Pesos por indicador

IndicadorPesoJustificación
EMA Cross30 ptsIndicador tendencial de mayor peso — confirma la dirección estructural
RSI25 ptsIdentifica extremos de precio — alta confiabilidad en zonas <35 / >65
MACD25 ptsConfirma momentum — histograma creciente/decreciente
Volumen20 ptsAmplificador — suma al bando dominante si hay spike de volumen

Feeds de mercado

Todos los datos del scanner son reales, obtenidos de APIs públicas de mercado. No hay datos simulados: si una fuente no responde, ese par no genera señal en ese ciclo y los demás continúan sin interrupción (Promise.allSettled).

Una sola fuente para todos los mercados: Yahoo Finance v8. Tanto cripto (feeds/crypto.js) como forex y futuros (feeds/forex.js) se conectan al mismo endpoint REST usando fetch nativo de Node — sin librerías externas ni API key. Cada feed mantiene su propio mapa de símbolos y un caché de 60 segundos por par para no exceder los límites de Yahoo (las velas M5 solo cierran cada 5 minutos, así que no tiene sentido pedir más seguido).

⚠️ En redes corporativas con proxy SSL es necesario setear NODE_TLS_REJECT_UNAUTHORIZED=0 en backend/.env, de lo contrario Yahoo Finance falla con error de certificado.

APIs externas

El scanner consume una única API pública de datos de mercado: Yahoo Finance v8. Cubre los tres mercados (cripto, forex y futuros) y no requiere autenticación.

APIUsada paraAccesoAuth

Endpoint Yahoo Finance v8

Endpoint REST público de gráficos. Se llama directamente con fetch y se mapea cada par interno a su símbolo de Yahoo. La respuesta trae timestamps e indicators.quote[0] (open/high/low/close/volume) que se transforman al formato OHLCV del scanner: [timestamp, open, high, low, close, volume]. 

// feeds/crypto.js y feeds/forex.js comparten esta misma llamada
const url =
  `https://query1.finance.yahoo.com/v8/finance/chart/${symbol}` +
  `?interval=${interval}&range=7d`

const res = await fetch(url, {
  headers: { 'User-Agent': 'Mozilla/5.0' }   // Yahoo rechaza sin User-Agent
})
const json = await res.json()
const result = json.chart.result[0]   // timestamps + indicators.quote[0]

Los timeframes del scanner se mapean a los intervalos de Yahoo: M1→1m, M5→5m, M15→15m.

ℹ️ Yahoo Finance no publica un contrato de API oficial; el endpoint v8 es de uso público pero no documentado formalmente. El caché de 60 s y el User-Agent minimizan el riesgo de rate-limiting. El precio de cripto en Yahoo (par -USD) puede diferir ligeramente del de Binance (par USDT) por ser de otra fuente.

WebSocket

El servidor expone un endpoint WebSocket en el mismo puerto que la API REST (ws://host:PORT en dev, wss://host en producción).

Comportamiento al conectarse

Al conectarse un nuevo cliente, el servidor envía inmediatamente el estado actual completo de todas las señales activas. Esto garantiza que la UI se popule sin esperar al próximo ciclo de escaneo.

Reconexión automática (frontend)

El hook useScanner.js implementa backoff exponencial:

1s → 2s → 4s → 8s → 16s → 30s (máximo)

El estado anterior de señales se preserva en memoria durante la reconexión — la tabla no se vacía.

Alertas Telegram

Cuando una señal nueva supera MIN_SCORE_ALERT (default 80), se envía un mensaje al chat configurado. Para evitar spam, hay un cooldown de 5 minutos por par.

Formato del mensaje

🟢 *BTC/USDT* — LONG
📊 Score: *80/100*
📐 Setup: RSI oversold + EMA Cross
💰 Precio: 67,842.50
⏱ Timeframe: M5
ℹ️ Si TELEGRAM_BOT_TOKEN no está configurado, las alertas se deshabilitan silenciosamente. El resto del sistema funciona con normalidad.

Base de datos

SQLite (better-sqlite3) en modo WAL. Se crea automáticamente en backend/signals.db al primer arranque.

Esquema

CREATE TABLE signals (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  pair      TEXT NOT NULL,
  market    TEXT NOT NULL,      -- 'crypto' | 'forex' | 'futures'
  direction TEXT NOT NULL,      -- 'LONG' | 'SHORT' | 'NEUTRAL'
  setup     TEXT NOT NULL,      -- descripción de indicadores activos
  score     INTEGER NOT NULL,
  price     REAL,
  timeframe TEXT NOT NULL,
  timestamp INTEGER NOT NULL    -- Unix ms
);

Confluencia Multi-Timeframe

Módulo independiente, accesible desde la pestaña "Confluencia" del frontend. Score firmado (−100 a +100)

A diferencia del score del scanner (0–100 con dirección aparte), aquí cada timeframe produce un score con signo: positivo = alcista, negativo = bajista. Usa los mismos pesos que scoring.js, pero sumándolos o restándolos según la dirección de cada indicador.

IndicadorAlcistaBajistaNeutral
EMA Cross+30−300
RSI+25−250
MACD+25−250
Volumen±20 al bando dominante0

El score general (overallScore) es el promedio de los 3 timeframes. La etiqueta de señal se deriva así:

overallScoreSeñal
≥ +70STRONG BUY
+40 a +69WEAK BUY
−39 a +39NEUTRAL
−40 a −69WEAK SELL
≤ −70STRONG SELL

Etiquetas de indicador por timeframe

Cada indicador se reporta con su estado textual: EMA/MACD como BULL / BEAR / NEUTRAL, RSI como OVERSOLD / OVERBOUGHT / NEUTRAL, y Volumen como HIGH / NORMAL.

Canal WebSocket

Sobre el mismo servidor WebSocket del scanner. El cliente se suscribe a un par y recibe un push cada 10 segundos. La suscripción es por cliente — no afecta a los demás.

// Cliente → servidor: suscribir
{ "type": "confluence_subscribe", "symbol": "BTC/USDT" }

// Cliente → servidor: desuscribir
{ "type": "confluence_unsubscribe" }

// Servidor → cliente: actualización (cada 10s)
{
  "type": "confluence_update",
  "timestamp": 1749200000000,
  "symbol": "BTC/USDT",
  "market": "crypto",
  "timeframes": {
    "M1":  { "ema": "BULL", "rsi": "OVERBOUGHT", "macd": "NEUTRAL", "volume": "NORMAL", "score": 5 },
    "M5":  { "ema": "BULL", "rsi": "OVERBOUGHT", "macd": "BULL",    "volume": "NORMAL", "score": 30 },
    "M15": { "ema": "BULL", "rsi": "NEUTRAL",    "macd": "BULL",    "volume": "NORMAL", "score": 55 }
  },
  "overallScore": 30,
  "signal": "NEUTRAL"
}
⚠️ El endpoint REST usa query param (?symbol=) en vez de path param porque el / de pares como BTC/USDT rompería la ruta.

Deploy en Railway

El proyecto usa un flujo GitHub → Railway autodeploy. Cada push a main dispara un nuevo deploy automáticamente.

Configuración (railway.toml)

[build]
builder      = "NIXPACKS"
buildCommand = "npm run build"   # instala deps + compila Vite

[deploy]
startCommand      = "NODE_ENV=production node backend/src/index.js"
healthcheckPath   = "/api/health"
healthcheckTimeout = 30

Qué hace el build

npm run build
→ npm install --prefix backend — instala Express, ws, etc.
→ npm install --prefix frontend — instala React, Vite, Tailwind
→ npm run build --prefix frontend — compila a frontend/dist/

En producción

El backend detecta NODE_ENV=production y sirve frontend/dist como archivos estáticos en /. El WebSocket usa automáticamente wss:// ya que el frontend deriva el protocolo de window.location.

💡 Un único servicio de Railway cubre todo: HTTP, WebSocket y archivos estáticos del frontend. No se necesita un servicio separado para el frontend.
