Todos los pares se procesan en paralelo con Promise.allSettled — si un par falla, los demás continúan. Solo se publican señales con score ≥ MIN_SCORE_PUBLISH (default 50), y solo se hace broadcast cuando el score o la dirección cambian respecto al ciclo anterior.

Confluencia multi-timeframe

feeds/confluence.js calcula M1/M5/M15 de un par reutilizando los feeds e indicadores. Score firmado por timeframe (mismos pesos que scoring): positivo alcista, negativo bajista. El overallScore promedia los tres.

overallScoresignal
≥ +70 / +40..+69STRONG BUY / WEAK BUY
−39..+39NEUTRAL
−40..−69 / ≤ −70WEAK SELL / STRONG SELL

Calculadora de posición

Lógica pura en usePositionCalc.js. Cálculos en base a entrada, % riesgo, distancia SL y RR:

monto en riesgo = capital × (%riesgo / 100)
distancia SL    = entrada × (%SL / 100)
SL  = LONG: entrada − distancia | SHORT: entrada + distancia
TP  = LONG: entrada + distancia×RR | SHORT: entrada − distancia×RR
tamaño = monto en riesgo / distancia SL

SL default por mercado: crypto 1.5% · forex 0.3% · futuros 0.5%. Cada cálculo se guarda en position_calculations al copiar el resumen.

Sesiones

sessions.js define horarios en UTC y expone funciones puras reutilizadas por la API, el scanner (is_high_liquidity) y el watcher WS.

sydney 21–6  ·  tokyo 0–9  ·  london 7–16  ·  newYork 12–21   (UTC)
overlaps: Tokyo/London 7–9 (medium) · London/NY 12–16 (high)
isHighLiquidity = overlap London/NY activo

El heatmap de volatilidad se calcula desde la tabla signals por hora UTC (densidad de señales), con fallback a datos típicos si no hay historial.

Briefing IA (Groq)

groq-client.js encapsula Groq (modelo llama-3.3-70b-versatile por default, configurable). Expone analyzeMarketContext, streamJSONCompletion (genérico) y checkGroqStatus. Toda llamada está envuelta en try/catch → error tipado groq_unavailable; nunca crashea.

El endpoint de análisis hace streaming por SSE: el cliente lo consume con fetch + reader (no EventSource, que no soporta POST). Rate limit de 30 min por sesión; el resultado se persiste en market_context.

data: {"type":"chunk","content":"…"}
data: {"type":"done","analysis":{ bias, key_levels, context_summary, key_events, scalping_tips, … }}

Journal

CRUD de operaciones + estadísticas SQL + coaching IA. Todo el PnL, RR y la sesión se calculan en el backend; las operaciones OPEN se excluyen de win rate y PnL.

PnL    = (LONG: exit−entry | SHORT: entry−exit) × tamaño
rr_actual = priceDiff / |entrada − SL|
result = WIN | LOSS | BREAKEVEN | OPEN

Análisis IA: mínimo 5 trades, rate limit 60 min, persistido en journal_analysis. Devuelve nota de disciplina, errores recurrentes, fortalezas, patrones emocionales, veredicto por setup y recomendaciones. Export CSV en GET /api/journal/trades/export.

Deploy en Railway

Flujo GitHub → Railway autodeploy: cada push a main dispara un deploy. No se deploya directo a Railway.

[build]
builder      = "NIXPACKS"
buildCommand = "npm run build"   # instala deps + compila Vite

[deploy]
startCommand       = "NODE_ENV=production node backend/src/index.js"
healthcheckPath    = "/api/health"

El backend detecta NODE_ENV=production y sirve frontend/dist como estáticos. Un único servicio cubre HTTP, WebSocket y frontend.
